# Module Collaboration Templates

这份文件用于把“模块协同”落成稳定模板。先定主模块，再定协作模块，不要平均用力。

## 1. 单模块模板

适用：

- 主要矛盾集中
- 只有一个维度明显主导结果

输出骨架：

```markdown
## 一句话判断
## 主问题分类
## 主模块
## 推荐路径
## 24小时动作
## 7天动作
## 风险与升级
```

## 2. 双模块模板

适用：

- 问题的“根”在一个模块
- 但执行成败明显受第二模块影响

推荐写法：

```markdown
## 一句话判断

## 主问题分类与模块
- 主模块：
- 协作模块：

## 主判断
- 主模块给出的根判断：

## 协作补充
- 协作模块补充的关键限制或放大器：

## 推荐路径

## 24小时动作

## 7天动作

## 关键话术

## 风险与升级
```

典型组合：

- 战略诊断 + 变革管理
- 组织效能 + 人际关系
- 绩效优化 + 人际关系
- 跨文化管理 + 组织效能

## 3. 三模块模板

适用：

- 战略、组织、关系或风险三条线互相缠绕
- 任何单模块判断都会偏

推荐写法：

```markdown
## 一句话判断

## 主问题分类与模块
- 主模块：
- 协作模块 A：
- 协作模块 B：

## 三层判断
- 方向层：
- 组织层：
- 关系/风险层：

## 推荐路径

## 24小时动作

## 7天动作

## 关键话术

## 失败触发点

## 何时升级外部支持
```

典型组合：

- 战略诊断 + 组织效能 + 变革管理
- 人际关系 + 组织效能 + 绩效优化
- 战略诊断 + 人际关系 + `escalation-boundaries.md`

## 4. 协同优先级规则

- 主模块负责定义问题
- 协作模块负责补充执行限制、情绪因素、组织承接或风险边界
- 如果协作模块开始改写主问题定义，应重新分类，而不是硬拼

## 5. 常见错误

- 一开始就上三模块，导致输出发散
- 模块都说一点，没人真正负责结论
- 协作模块写成背景知识，而没有进入决策
- 高风险问题没有单独拉出 `escalation-boundaries.md`
